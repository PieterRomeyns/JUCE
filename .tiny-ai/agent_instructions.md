## 3. How JUCE Is Integrated (Local Submodule)

> [!IMPORTANT]
> JUCE is **not** fetched from the internet. It is a **local git submodule** at `_deps/juce-submodule/`. This is already cloned and checked out.

### 3.1 Submodule Setup

The `.gitmodules` file defines:
```ini
[submodule "_deps/juce-submodule"]
    path = _deps/juce-submodule
    url = https://github.com/juce-framework/JUCE.git
```

The submodule is currently at tag `8.0.12` (commit `0ae64bf506`). The root `CMakeLists.txt` references `JUCE_VERSION "8.0.7"` but the actual checked-out code is newer — this is fine, they are API-compatible.

### 3.2 How It Connects to CMake

The root `CMakeLists.txt` includes JUCE via:
```cmake
add_subdirectory(_deps/juce-submodule)
```

This makes all JUCE CMake targets available globally:
- `juce::juce_core`
- `juce::juce_audio_processors`
- `juce::juce_gui_basics`
- `juce::juce_audio_basics`
- `juce::juce_audio_devices`
- `juce::juce_audio_formats`
- `juce::juce_audio_utils`
- `juce::juce_dsp`
- `juce::juce_gui_extra`
- `juce::juce_data_structures`
- `juce::juce_events`
- `juce::juce_graphics`
- `juce::juce_recommended_config_flags`
- ... and more (see `_deps/juce-submodule/modules/` for full list)

### 3.3 How to Use JUCE Headers in C++ Code

There are **two** include styles used in this project. Be consistent with the file you're editing:

**Style A — Module includes (preferred for library targets):**
```cpp
#include <juce_core/juce_core.h>
#include <juce_audio_processors/juce_audio_processors.h>
```

**Style B — Unified header (used in app-level code):**
```cpp
#include <JuceHeader.h>
```

`JuceHeader.h` is auto-generated by `juce_generate_juce_header(MyDAW)` in the root CMakeLists. It is only available to the `MyDAW` target. Library targets (`Effects`, etc.) should use **Style A**.

### 3.4 If the Submodule Is Missing or Empty

If `_deps/juce-submodule/` is empty (e.g., after a fresh clone), run:
```bash
git submodule update --init --recursive
```

## 4. CMake Build System Architecture

### 4.1 Root CMakeLists.txt (the most important file)

```cmake
cmake_minimum_required(VERSION 3.14)
project(MyDAW VERSION 1.0 LANGUAGES C CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED True)

enable_testing()

add_subdirectory(_deps/juce-submodule)   # ← Makes JUCE available
add_subdirectory(Source)                  # ← Delegates to Source/CMakeLists.txt
add_subdirectory(tests)                   # ← Delegates to tests/CMakeLists.txt

# Main app target
juce_add_gui_app(MyDAW PRODUCT_NAME "MyDAW")
juce_generate_juce_header(MyDAW)

target_sources(MyDAW PRIVATE
    Source/Main.cpp
    Source/UI/MainWindow.cpp)

target_compile_definitions(MyDAW PRIVATE
    JUCE_WEB_BROWSER=0
    JUCE_USE_CURL=0
    JUCE_APPLICATION_NAME_STRING="$<TARGET_PROPERTY:MyDAW,JUCE_PRODUCT_NAME>"
    JUCE_APPLICATION_VERSION_STRING="$<TARGET_PROPERTY:MyDAW,JUCE_VERSION>"
    JUCE_MAC)

target_link_libraries(MyDAW
    PRIVATE juce::juce_core juce::juce_gui_basics juce::juce_audio_processors
    PUBLIC  juce::juce_recommended_config_flags)
```

### 4.2 Per-Module CMakeLists (Example: Effects)

Each subdirectory under `Source/` has its own `CMakeLists.txt` that creates a static library:

```cmake
# Source/Effects/CMakeLists.txt
cmake_minimum_required(VERSION 3.14)
project(Effects)

add_library(Effects STATIC
    DelayFX.cpp
    # Add new .cpp files here as you create them
)

target_include_directories(Effects PUBLIC 
    ${CMAKE_CURRENT_SOURCE_DIR}
    ${CMAKE_CURRENT_SOURCE_DIR}/../..
)

target_link_libraries(Effects PUBLIC
    juce::juce_audio_processors
    juce::juce_core
)
```

### 4.3 How to Add a New Source File

**Step 1:** Create the `.h` and `.cpp` files in the appropriate `Source/` subdirectory.

**Step 2:** Add the `.cpp` file to the corresponding `CMakeLists.txt`:
```cmake
# Example: adding ChorusFX to Effects
add_library(Effects STATIC
    DelayFX.cpp
    ChorusFX.cpp        # ← Add here
)
```

**Step 3:** If the new code needs JUCE modules not already linked, add them:
```cmake
target_link_libraries(Effects PUBLIC
    juce::juce_audio_processors
    juce::juce_core
    juce::juce_dsp           # ← Add if needed
)
```

**Step 4:** If the main `MyDAW` app target uses the new code, link the library:
```cmake
# In root CMakeLists.txt
target_link_libraries(MyDAW PRIVATE
    juce::juce_core
    juce::juce_gui_basics
    juce::juce_audio_processors
    Effects                   # ← Add library target
)
```

### 4.4 How to Add a New Subdirectory Module

If you create a new subdirectory (e.g., `Source/Audio/`):

1. Create `Source/Audio/CMakeLists.txt` with the library definition
2. Add `add_subdirectory(Audio)` to `Source/CMakeLists.txt`
3. Link the new library wherever needed

### 4.5 Empty Modules

`Source/Engine/`, `Source/Instruments/`, and `Source/Model/` currently have skeleton CMakeLists with no source files. When you add files, update them to follow the `Effects` pattern.

## 5. C++ Code Conventions

### 5.1 General Style

- **C++ Standard:** C++17 (no C++20 features)
- **Naming:** PascalCase for classes and types, camelCase for variables and methods
- **Smart pointers:** Use `std::unique_ptr` or `std::shared_ptr` — never raw owning pointers
- **Include guards:** Use `#pragma once`
- **JUCE macros:** End every class with `JUCE_DECLARE_NON_COPYABLE_WITH_LEAK_DETECTOR(ClassName)`

### 5.2 AudioProcessor Pattern (Effects / Instruments)

Every effect and instrument is a `juce::AudioProcessor` subclass. Follow this pattern:

```cpp
// ChorusFX.h
#pragma once
#include <juce_audio_processors/juce_audio_processors.h>
#include <juce_core/juce_core.h>

class ChorusFX : public juce::AudioProcessor
{
public:
    ChorusFX();
    ~ChorusFX() override;

    const juce::String getName() const override { return "Chorus FX"; }
    void prepareToPlay(double sampleRate, int samplesPerBlock) override;
    void releaseResources() override;
    void processBlock(juce::AudioBuffer<float>& buffer, juce::MidiBuffer& midi) override;

    // Required AudioProcessor overrides (stubs for non-plugin usage):
    bool acceptsMidi() const override { return false; }
    bool producesMidi() const override { return false; }
    double getTailLengthSeconds() const override { return 0.0; }
    int getNumPrograms() override { return 1; }
    int getCurrentProgram() override { return 0; }
    void setCurrentProgram(int) override {}
    const juce::String getProgramName(int) override { return {}; }
    void changeProgramName(int, const juce::String&) override {}
    bool hasEditor() const override { return false; }
    juce::AudioProcessorEditor* createEditor() override { return nullptr; }
    void getStateInformation(juce::MemoryBlock&) override {}
    void setStateInformation(const void*, int) override {}

    // APVTS for parameter management
    juce::AudioProcessorValueTreeState apvts;

private:
    // DSP state — pre-allocate in prepareToPlay()
    
    JUCE_DECLARE_NON_COPYABLE_WITH_LEAK_DETECTOR(ChorusFX)
};
```

> [!WARNING]
> `juce::AudioProcessor` has many pure virtual methods. If you forget the stubs listed above, the class will be abstract and won't compile. Always include all required overrides.

### 5.3 APVTS (AudioProcessorValueTreeState) Usage

Parameters are managed via APVTS. Create the parameter layout in the constructor:

```cpp
ChorusFX::ChorusFX()
    : apvts(*this, nullptr, "Parameters",
        {
            std::make_unique<juce::AudioParameterFloat>(
                juce::ParameterID("rate", 1), "Rate",
                juce::NormalisableRange<float>(0.1f, 5.0f, 0.01f), 1.0f),
            std::make_unique<juce::AudioParameterFloat>(
                juce::ParameterID("depth", 1), "Depth",
                juce::NormalisableRange<float>(0.0f, 1.0f, 0.01f), 0.5f),
            std::make_unique<juce::AudioParameterFloat>(
                juce::ParameterID("mix", 1), "Mix",
                juce::NormalisableRange<float>(0.0f, 1.0f, 0.01f), 0.5f),
        })
{
}
```

Read parameters in `processBlock()`:
```cpp
float rate = apvts.getRawParameterValue("rate")->load();
```

### 5.4 Real-Time Safety Rules

Inside `processBlock()` and the audio callback (`audioDeviceIOCallbackWithContext`):

| ❌ Forbidden | ✅ Use Instead |
|---|---|
| `new` / `delete` / `malloc` | Pre-allocate in `prepareToPlay()` |
| `std::mutex`, `std::lock_guard` | `std::atomic`, `juce::AbstractFifo` |
| File I/O, logging, `printf` | Do it on the message thread |
| `std::vector::push_back` (that grows) | Pre-sized containers |
| `juce::String` construction | Pre-built strings |

## 7. Testing Conventions

### 7.1 Test File Pattern

Tests live in `tests/`. Each test file is named `<ClassName>Test.cpp`. Example:

```cpp
// tests/ChorusFXTest.cpp
#include "ChorusFX.h"
#include <juce_core/juce_core.h>

int main()
{
    // Test 1: at mix=0, output equals input
    ChorusFX processor;
    processor.prepareToPlay(44100.0, 128);
    
    juce::AudioBuffer<float> buffer(2, 128);
    // Fill with known signal...
    
    juce::MidiBuffer midi;
    processor.processBlock(buffer, midi);
    
    // Assert output matches expected...
    // Return 0 on success, non-zero on failure
    return 0;
}
```

### 7.2 Registering Tests in CMake

Add each test to `tests/CMakeLists.txt`:

```cmake
# Add test executable
add_executable(ChorusFXTest ChorusFXTest.cpp)

# Link with the library containing the class under test
target_link_libraries(ChorusFXTest PRIVATE 
    Effects
    juce::juce_core
    juce::juce_audio_processors
)

# Register with CTest
add_test(NAME ChorusFXTest COMMAND ChorusFXTest)
```

> [!NOTE]
> The existing `DelayFXTest.cpp` uses Catch2 headers (`#include "catch2/catch_test_macros.hpp"`) but Catch2 is **not** set up as a dependency. Newer tests should use the simple `main()` + return-code pattern shown above, unless you add Catch2 to the build. The existing test also references `"JUCEHeader.h"` which is only available in the MyDAW target — this test may not compile as-is.

### 8.3 PhaserFX.h Uses Incorrect Method Signatures

`Source/Effects/PhaserFX.h` declares methods like `getParameters()`, `getParameterNames()`, `getParameterLabels()`, `getParameterUnits()` that are not part of the `juce::AudioProcessor` API in JUCE 8. This header needs to be rewritten to match the standard AudioProcessor pattern (see Section 5.2).

### 8.4 Missing AudioProcessor Stubs

Both `DelayFX` and `PhaserFX` are missing required `juce::AudioProcessor` pure virtual overrides (`acceptsMidi`, `producesMidi`, `getTailLengthSeconds`, `getNumPrograms`, etc.). These will cause linker errors or abstract-class errors.

## 10. Critical Do's and Don'ts

### ✅ DO

- Read `design.md` for exact class interfaces before implementing
- Pre-allocate all DSP buffers in `prepareToPlay()`, not in `processBlock()`
- Use `std::atomic` for state shared between audio and UI threads
- Use `juce::LinearSmoothedValue` for parameter changes (avoid zipper noise)
- Use `juce::jlimit`, `juce::jmax`, `juce::jmin` for clamping
- Build and test after every change
- Add a CTest unit test for every non-trivial class
- Keep the existing `auval` plugin validation passing

### ❌ DON'T

- Modify anything in `.tiny-ai/` unless explicitly asked
- Use Objective-C (`#import`, `@interface`, `NSObject`, AppKit, Cocoa)
- Use `new`/`delete`/`malloc` on the audio thread
- Use `std::mutex` or any blocking lock on the audio thread
- Introduce new third-party libraries without asking
- Skip tasks or implement features not in the spec
- Present code that hasn't been compiled and tested
- Modify `DSP/*.h`, `MyAmpPlugin.cpp`, or `MyAmpPlugin.h` (if they exist)

## 12. Quick Reference: Common JUCE Patterns

### Audio Device Management
```cpp
juce::AudioDeviceManager deviceManager;
deviceManager.initialiseWithDefaultDevices(2, 2); // 2 in, 2 out
```

### Parameter Smoothing (Real-Time Safe)
```cpp
juce::LinearSmoothedValue<float> volume;
// In prepareToPlay:
volume.reset(sampleRate, 0.02); // 20ms smoothing
// In processBlock:
volume.setTargetValue(newValue);
for (int i = 0; i < numSamples; ++i) {
    sample *= volume.getNextValue();
}
```

### Timer-Driven UI Repaints
```cpp
class MyComponent : public juce::Component, public juce::Timer
{
    void timerCallback() override { repaint(); }
    // In constructor: startTimerHz(30);
};
```

### Thread-Safe Audio→UI Communication
```cpp
juce::AbstractFifo fifo(1024);
// Audio thread: write peak levels into FIFO
// UI thread: read from FIFO in timerCallback
```

### File I/O (Background Thread Only)
```cpp
juce::WavAudioFormat wav;
auto writer = wav.createWriterFor(outputStream.get(), sampleRate, numChannels, 16, {}, 0);
```
